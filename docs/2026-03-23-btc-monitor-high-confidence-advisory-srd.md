# BTC Monitor SRD: High-Confidence BTC Advisory Review

> **For Claude Code:** This document is the source of truth for reviewing `btc-monitor` as a high-confidence BTC add/reduce advisory system. The architecture is now stateless and advisory-only, but the current backtest precision is not strong enough for high-confidence use. Budget, capital sizing, and monthly rollover logic are out of scope for the target system.

## Metadata

- Status: Review complete, final advisory architecture verified, precision work still required
- Date: 2026-03-23
- Audience: Maintainers and Claude Code agents
- Scope: Current-system evaluation, factor completeness audit, backtest credibility audit, refactor requirements, and acceptance criteria

## Acceptance Decision

The current system is **not accepted** for the target use case of high-confidence BTC add/reduce advice.

It is acceptable as a stateless advisory framework, but it does not yet meet the standard for high-confidence position guidance because:

- the measured action precision is too low for the intended use case
- confidence saturates instead of discriminating strong versus weak signals
- the current validation emphasizes short-horizon precision more than BTC cycle reality
- the factor set is still too proxy-heavy for reliable timing
- the advisory engine needs stronger calibration and horizon-aware acceptance gates

## Objective

Build a decision-support engine that produces **advice only**, not execution, and that is calibrated against historical advisory quality:

- `ADD`
- `REDUCE`
- `HOLD`
- `INSUFFICIENT_DATA`

Each recommendation must include:

- confidence score
- factor evidence summary
- missing-data blockers
- conflict flags
- source freshness
- calibration context

## Non-Goals

- no budget multiplier
- no accumulated monthly cash logic
- no stateful position sizing
- no auto-execution
- no paid APIs or paid datasets
- no claim of exact market top or bottom timing

## Current System Strengths

The current repository has several sound building blocks:

- layered separation exists between strategic, tactical, and advisory logic
- the live path is stateless and advisory-only
- research-only factors are excluded from production scoring
- the report surfaces coverage, missing core factors, and excluded research factors
- the backtest is now authentic and uses real historical data
- unit-test coverage is good for a small project

These strengths are useful, but they do not offset the issues below.

## Critical Findings

### 1. Precision is too low for high-confidence advice

The current advisory backtest reports:

- `ADD` precision: `1.03%`
- `REDUCE` precision: `1.52%`
- history length: `450` weeks
- action mix: `HOLD 62.67%`, `ADD 22.67%`, `REDUCE 14.67%`

That is far below what a user would reasonably call high-confidence advice. The engine is now operationally clean, but the signal quality is not yet strong enough.

### 2. Confidence is not calibrated yet

The current confidence output saturates at `100` in the real backtest run.

That means confidence is not currently expressing relative conviction across the history window. It is a thresholded pass/fail score, not a calibrated confidence estimate.

### 3. The chosen evaluation horizon is too short to characterize BTC cycle advice

The current backtest emphasizes 28-day forward precision. That is useful as a diagnostic, but BTC add/reduce advice is cycle-sensitive and often needs longer horizons to evaluate fairly.

As a result, the current backtest can understate the quality of slower cycle calls and can also overrate tactical false positives if the market moves later than 4 weeks.

### 4. The factor mix is still proxy-heavy

The current system still leans on hand-shaped proxy thresholds for:

- `MVRV_Proxy`
- `Puell_Multiple`
- `200WMA`
- `Net_Liquidity`
- `Yields`

That is reasonable for a first-pass advisory framework, but it is not enough to produce robust high-confidence timing by itself. The engine needs at least one more independent recovery/timing lens and one stronger macro risk lens.

### 5. The tactical layer is still too coarse for precise reduce calls

The tactical layer is functional, but it remains a small confirmation layer rather than a rich short-horizon model. That means the system still lacks a strong way to distinguish:

- a clean cyclical pullback
- a noisy local bounce
- a genuine tactical overheating event

This is part of why `REDUCE` precision remains low.

## Factor Completeness Audit

### Current factors that should remain

- `Puell_Multiple`
  - independent and economically meaningful
- `200WMA`
  - useful as a slow trend anchor
- `Net_Liquidity`
  - important macro liquidity proxy
- `Yields`
  - useful macro tightening proxy if redefined as regime rather than one-week delta
- `FearGreed`
  - acceptable as a tactical sentiment input
- `RSI_Div`
  - acceptable as a tactical confirmation input

### Current factors that should remain but be downgraded in confidence

- `MVRV_Proxy`
  - useful, but it is still a proxy based on a 730-day moving average rather than true realized-cap data
- `Cycle_Pos`
  - useful as supporting context, but it is too correlated with price-location factors to carry full strategic weight by itself

### Factors that should be removed from the live production path or formalized as research-only

- `Production_Cost`
  - keep research-only until a durable free-data model exists
- `Options_Wall`
  - keep research-only until both live and historical data are stable and replayable
- `ETF_Flow`
  - keep research-only until a free historical source is formalized

### Missing factors needed for high-confidence add/reduce advice

The target engine needs at least one additional factor in each of the blocks below, and it needs a calibrated way to value them.

### A. Stronger independent bottom/recovery factor

Recommended addition:

- `Hash_Ribbon` or equivalent miner-stress recovery signal using free hash-rate and difficulty data

Reason:

- this adds a bottom/recovery lens that is not just another price-location proxy

### B. Broader macro risk block

Recommended addition:

- `DXY_Regime` from FRED
- or `10Y_Real_Yield_Regime` if a robust free source is available

Reason:

- `Net_Liquidity` and nominal 10Y yield are not enough for high-confidence macro filtering

### C. Tactical stretch factor

Recommended addition:

- short-term stretch or reversion percentile
- example: 4-week price stretch versus 200DMA or 26-week trend anchor

Reason:

- this makes `STAGGER_BUY` or intermediate tactical states reachable without over-relying on RSI divergence

### D. A formal risk-off confirmation block for reduce calls

Recommended addition:

- one free and replayable market-structure factor, such as basis/funding/open-interest regime, only if it satisfies free-history and stability requirements

Reason:

- reduce calls should not be triggered by one overheated price-location factor alone

### Factor policy for the target system

The target system should enforce block-level independence:

- `ADD` requires bullish evidence from at least three independent blocks
- `REDUCE` requires bearish evidence from at least two independent blocks, with at least one non-price-location block
- missing required blocks must return `INSUFFICIENT_DATA`
- confidence must be calibrated from historical bucket precision, not from factor count alone
- 28-day precision should be treated as diagnostic; 84-day and 182-day precision should carry more acceptance weight

## Backtest Credibility Audit

### What is credible today

- signal timing mostly avoids same-bar execution bias
- live/backtest parity tests exist and pass
- research-only factors are invalidated in backtest instead of being silently guessed

### What is not credible enough for the target use case

### 1. Backtest semantics change when data is missing

Macro history depends on `FRED_API_KEY`.

If macro data is unavailable:

- `Net_Liquidity` and `Yields` become invalid
- the strategy still runs
- the meaning of the strategic score changes

For a high-confidence system, missing required macro blocks must fail closed.

### 2. The tactical layer degrades in backtest

Current backtest facts from the current code:

- confidence saturates at `100`
- action distribution is not too sparse, but the quality of the non-hold actions is poor
- 28-day precision is too low to support strong advisory claims

This shows that the backtest is now real, but the decision layer still needs calibration.

### 3. The backtest optimizes the wrong thing for the target product

Current backtest output is advisory-oriented, but the current acceptance emphasis is still too short-horizon:

- 28-day precision
- action distribution
- regime averages

The target backtest must measure:

- forward 4-week, 12-week, and 26-week returns after each recommendation
- precision of `ADD`
- precision of `REDUCE`
- false-positive and false-negative rates
- regime lag around cycle turns
- monotonicity of confidence buckets
- calibration curves by confidence bucket and horizon

### 4. Full-history single-run evaluation is not enough

Bitcoin has multiple structurally different regimes. A single full-history run hides whether the strategy only works because of one favorable segment.

The target validation should include:

- cycle-sliced evaluation
- walk-forward windows
- regime-conditioned performance tables

## Target Product Requirements

### Product behavior

The target engine must emit:

- `ADD`
- `REDUCE`
- `HOLD`
- `INSUFFICIENT_DATA`

The output payload must include:

- confidence score from 0 to 100
- strategic state
- tactical state
- ordered supporting factors
- ordered conflicting factors
- missing required factors
- source timestamps
- explanation text for humans
- calibration metadata or references to calibration buckets

### Decision rules

The target engine must not map score directly to action.

It must use rule-based gating:

- `ADD` requires minimum coverage and agreement across independent evidence blocks
- `REDUCE` requires explicit overheating or deterioration agreement across independent blocks
- conflicting evidence lowers confidence
- missing required data returns `INSUFFICIENT_DATA`
- confidence must be derived from both current evidence and historical calibration, not only from the current factor count

### Architecture requirements

Recommended module split:

- `src/strategy/factor_registry.py`
  - source of truth for factor metadata
- `src/strategy/strategic_engine.py`
  - slow regime inference only
- `src/strategy/tactical_engine.py`
  - short-horizon confirmation only
- `src/strategy/advisory_engine.py`
  - final advice and confidence
- `src/strategy/reporting.py`
  - advice report only
- `src/backtest/advisory_backtest.py`
  - advisory-quality validation only

### Reporting requirements

The report must make blocked decisions explicit:

- `coverage failed`
- `required macro block missing`
- `reduce signal blocked by missing confirmation`
- `research factor excluded`

The report must not imply that a recommendation is actionable if the engine is missing required evidence.

The report must also show:

- confidence bucket
- calibration context for the current horizon
- whether the recommendation is being validated on a short, medium, or long horizon

## Recommended Refactor Sequence

1. Calibrate confidence against historical precision buckets.
2. Promote 84-day and 182-day validation to primary acceptance checks.
3. Add one stronger non-price recovery factor and one broader macro factor.
4. Strengthen tactical confirmation so reduce calls are less noisy.
5. Reweight or downgrade overly correlated proxy factors.
6. Regenerate and recommit backtest artifacts only after the new schema is stable.
7. Align `README.md`, `GEMINI.md`, and `src/config.py` with actual runtime behavior.

## Acceptance Criteria

The refactor is accepted only when all of the following are true:

- the core engine emits `ADD`, `REDUCE`, `HOLD`, or `INSUFFICIENT_DATA`
- `ADD` and `REDUCE` both fail closed when required factor blocks are missing
- a one-factor bullish scenario cannot produce `ADD`
- a one-factor overheated scenario cannot produce `REDUCE`
- confidence does not saturate to a single value across the full backtest
- confidence bucket monotonicity is documented and testable
- advisory backtest reports forward-return precision for 28-day, 84-day, and 182-day horizons
- cycle-sliced or walk-forward validation is documented
- README and repo docs no longer claim calibration or precision properties unless they are implemented in code
- Docker test suite passes

## Current Status Against Acceptance Criteria

Current status: **Rejected**

Reasons:

- precision is too low for high-confidence advice
- confidence is not calibrated
- the target horizon needs stronger multi-window validation
- the factor set needs stronger independent recovery and macro evidence

## Target Improvement Summary

The system should evolve toward:

- calibrated confidence, not score saturation
- horizon-aware validation, not 28-day-only judgment
- stronger factor independence, not proxy accumulation
- explicit conservative failure, not optimistic certainty

## Deliverable of This Review

This SRD is the acceptance report for the current system and the source document for the next refactor.

It does not approve the current implementation for high-confidence add/reduce advice, but it does confirm the architecture is now advisory-only and the remaining gap is signal quality.
